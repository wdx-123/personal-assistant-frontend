## #5 前端分层与实现顺序

1. 新功能默认按以下顺序推进：
   - `types/constants`
   - `services`
   - `stores/composables`
   - `views/components`
   - `router/permission`
   - `verification`
2. 各层职责固定如下：
   - `src/types`：领域类型、接口类型、通用响应类型
   - `src/constants`：状态码、校验规则、共享常量
   - `src/services`：纯 API 调用和参数拼装
   - `src/stores`：跨页面状态、登录态、权限态、组织态、业务编排
   - `src/composables`：可复用的页面逻辑或业务交互逻辑
   - `src/components/common`：通用无业务组件
   - `src/components/business`：领域业务组件
   - `src/components/layout`：布局组件
   - `src/views`：页面级编排与交互
   - `src/router`：路由定义、守卫、路由映射
3. 简单纯 UI 改动可以跳过无关层，但不得把 API、持久化、权限或路由逻辑直接塞回页面组件。
4. 新增接口时，先补类型，再补 service，再决定是否需要 store/composable 承接，不允许反向从页面直接倒推请求实现。
5. 默认要做：
   - 优先复用 `components/common`、`components/business`、现有 store、现有 request 能力
   - 新增管理台页面时优先评估 `ant-design-vue` 的表格、表单、弹窗
   - 新增首页、登录、品牌展示类界面时优先走自定义组件和 Tailwind

## #6 页面、组件与 composable 规则

1. 页面组件负责页面级编排，包括：
   - 路由参数处理
   - 页面级异步加载
   - 页面级状态展示
   - 组合多个业务组件
2. 页面不负责沉淀可复用业务逻辑；复用逻辑应下沉到 composable 或 store。
3. 组件拆分原则：
   - 可跨页面复用的基础交互放 `components/common`
   - 有明确业务语义的交互放 `components/business`
   - 页面骨架、导航、侧栏、头部放 `components/layout`
4. `composables` 适用于以下场景：
   - 多页面共享逻辑
   - 单组件内逻辑过长、但又不应提升为全局状态
   - 包含副作用、计时器、事件监听、数据拉取协调的复用逻辑
5. 新增组件默认要求：
   - 组件文件使用 `PascalCase.vue`
   - 优先用 `props + emits` 建立边界
   - 明确处理 `loading / empty / error / disabled`
   - 有副作用时在卸载时清理
6. 新增或大改文件默认向以下目标收敛：
   - 单文件尽量控制在 300 行以内
   - 单个函数尽量控制在 50 行以内
   - 嵌套层级尽量不超过 3 层
7. 若因为历史原因暂时无法拆分，必须优先保证职责清晰，而不是继续堆叠页面级业务流程。

## #7 Store 规则

1. Store 负责全局状态、派生状态、持久化状态与跨页面业务编排。
2. 以下状态统一由 store 承接，不在组件中直接持久化：
   - 登录态
   - Token
   - 菜单权限
   - 浏览组织
   - 其它跨页面共享状态
3. Store 可以调用 service 做业务编排，但不得绕过 service 直接操作 axios。
4. Store 应优先使用 Setup Store 写法，并通过 `computed` 提供派生状态。
5. 仅当前页面局部使用、离开页面即失效的状态，不要提升到全局 store。
6. 组件不得直接写登录态、菜单、组织等本地持久化；这类逻辑必须收口到 store 或既有请求基础设施。

## #8 Service 与请求链路规则

1. 页面、组件、composable、store 都禁止直接调用裸 axios；统一通过 `src/services/*` 和 `src/utils/request.ts`。
2. `src/services/*.service.ts` 只负责：
   - 定义类型化接口函数
   - 组织 URL、方法、参数、请求选项
   - 返回业务数据
3. `service` 层禁止承担以下职责：
   - 弹出成功或失败提示
   - 跳转路由
   - 写本地持久化
   - 直接操作 DOM
4. 请求链路以 `src/utils/request.ts` 为统一入口，以下能力由其负责：
   - 通用成功提示和失败提示
   - Token 注入
   - Token 刷新与重试
   - 通用错误归一化
5. 页面、store、业务流程通过以下扩展选项控制提示策略：
   - `skipTip`
   - `skipErrTip`
   - `skipSuccTip`
   - `customErrTip`
   - `customSuccTip`
6. 新增接口必须满足：
   - 先补充类型定义
   - service 返回明确类型
   - 参数命名、分页字段、状态码处理与现有接口风格一致
7. 禁止事项：
   - 禁止在组件内 `axios.create`
   - 禁止直接请求接口绕过 service
   - 禁止写死接口地址、状态码文案或重复封装第二套 request 基础层

## #9 Router 与权限规则

1. 路由定义统一收口在 `src/router/routes.ts`。
2. 路由守卫与认证检查统一收口在 `src/router/guards.ts`。
3. 动态权限路由和菜单权限统一通过 `src/stores/permission.ts` 生成与过滤。
4. 路由映射、菜单路径映射等前后端协同信息，应维护在 `src/router` 相关文件，不要分散硬编码到页面组件。
5. 新增受控页面时，必须同步评估并必要时更新：
   - 路由定义
   - `meta.title`
   - redirect 逻辑
   - 菜单映射或路径映射
   - 权限过滤逻辑
   - 侧边栏展示与激活行为
6. 不要把动态权限项目误当成纯静态菜单项目来改。
7. 路由标题、登录跳转、404、动态注入链路修改后，必须做一次实际导航验证。

## #10 UI 体系与样式规则

1. UI 体系优先级：
   - `Tailwind + 自定义组件`
   - `ant-design-vue` 仅用于复杂表单、表格、弹窗、管理台交互
2. 在同一页面内必须保持视觉体系一致，避免无理由混用多套按钮、输入框、表格和反馈样式。
3. 简单布局、间距、排版、响应式优先使用 Tailwind。
4. 复杂动画、3D 翻转、特殊交互可使用 `scoped CSS`，但要控制作用域并避免样式外溢。
5. 新增色值、阴影、圆角、断点、动效时，优先复用 `tailwind.config.js` 和现有设计令牌；不要随手硬编码。
6. 首页、登录页、品牌展示页优先保持项目现有自定义视觉语言，不为了“统一”而强行改成管理台风格。
7. 管理台中需要快速交付且适合现有依赖时，优先复用 `ant-design-vue`，不要重复封装已有成熟能力。
8. 禁止事项：
   - 不同时维护两套同职能组件
   - 不新增未经确认的 UI 框架
   - 不滥用内联样式替代可维护的类名体系

## #11 TypeScript 与命名规则

1. 禁止新增 `any`；拿不准的未知错误或外部数据优先使用 `unknown` 后再收窄。
2. 禁止无说明地使用 `@ts-ignore` 或 `@ts-expect-error`；若必须使用，必须注明原因和移除条件。
3. 禁止新增魔法数字、硬编码接口地址、硬编码状态码文案、硬编码颜色与断点。
4. 命名规则：
   - 组件文件：`PascalCase.vue`
   - 工具文件：`camelCase.ts`
   - 服务文件：`camelCase.service.ts`
   - 类型文件：`camelCase.types.ts`
   - 常量：`UPPER_SNAKE_CASE`
   - 布尔值：`is/has/should/can` 前缀
5. 类型定义优先使用当前项目既有目录与命名方式，不额外发明平行体系。
6. 新增类型时优先提升复用价值，避免同一业务对象在多个文件中重复定义近似结构。
7. 现有遗留 `any` 可在触达相关代码时逐步消除，但新增和修改部分不得继续扩散。

## #12 性能与 UX 规则

1. 所有页面级异步操作必须显式处理：
   - `loading`
   - `empty`
   - `error`
   - `disabled`
   - `retry`
2. 所有提交类操作都要防重复提交。
3. 重型页面、重型业务组件、路由页面默认优先考虑懒加载。
4. 使用定时器、事件监听、轮询、路由离场动画或外部订阅时，必须在合适时机清理副作用。
5. 列表渲染必须使用稳定唯一 key，不要使用 index 作为长期 key。
6. 在 `v-if` 与 `v-show` 间做选择时，以真实交互频率和渲染成本为依据，不机械套用。
7. 所有新增页面都要兼顾桌面端和移动端可用性，至少完成基本响应式与溢出检查。
8. 优先优化用户能感知到的体验，包括：
   - 加载反馈
   - 错误提示
   - 禁用态
   - 切换态
   - 关键操作成功反馈
9. 不做为了炫技而增加复杂实现；性能优化必须服务于真实交互收益。

## 计划落盘规则

1. 只要任务属于新增、重构、修复、联调、排障、迁移、删除、配置调整这类执行型工作，先写计划，不直接改代码。
2. 计划文件固定写到 `plan/<module>/pending-<task>.md`。
3. 结构名固定使用英文：根目录为 `plan/`，跨模块目录为 `plan/cross-module/`，状态前缀为 `pending-` 和 `approved-`。
4. `<module>` 和 `<task>` 按语义决定中英文：稳定技术名词优先英文，如 `permission`、`home`；更自然的业务表达可保留中文，如 `首页`、`卡片布局调整`。
5. 纯问答、纯解释、纯代码审查、纯只读排查，不强制生成计划文件。
6. 计划目录规则以 `plan/README.md` 为准。

## 计划命名规则

1. 文件名格式固定为 `pending-<task>.md` 或 `approved-<task>.md`。
2. `<task>` 必须直接体现本次执行目标，可用英文技术短语，也可用中文业务短语，但都不能空泛。
3. 合格示例：`pending-dynamic-route-cleanup.md`、`pending-卡片布局调整.md`、`pending-组织权限联调.md`。
4. 后续若用户直接说“先出计划”，默认先写入对应模块目录下的 `pending-<task>.md`，无需额外指定路径。

## 审查后执行规则

1. 生成待审计划后，只允许查代码、读文档、跑非修改型检查；未获明确确认前，不允许实施改动。
2. 用户明确确认后，先将计划文件改名为 `approved-<task>.md`，再按计划执行。
3. 执行前需要在对话中回报计划路径和摘要，供用户审查。
4. 若执行中发现范围明显变化，禁止静默扩项，必须重新生成新的 `pending-<task>.md` 给用户复审。
5. 涉及页面、路由、权限、请求链路或接口联调的执行任务，同样必须先经过待审计划流程。

## #14 提问引用规则

1. 向用户提澄清问题时，必须在问题后追加一行：
   - `依据规则: #<rule-number>`
2. 若同时依据多条规则，使用逗号分隔：
   - `依据规则: #5,#8,#9`
3. 若你是基于默认假设继续推进，也应在说明中明确写出假设，不要把假设伪装成事实。

## 开发时默认遵守的明确边界

### 要做

- 优先读现有代码，再决定是否新增类型、service、store、composable、组件或路由
- 优先复用 `components/common`、`components/business`、`stores`、`utils/request.ts`
- 页面发起异步操作时显式维护状态和失败处理
- 新增接口时先补类型，再补 service，再接入上层
- 新增受权限控制页面时同步检查路由、菜单映射、侧边栏和守卫链路

### 不做

- 不在组件里直接创建 axios 实例或直接请求接口
- 不直接在组件里写鉴权持久化、菜单持久化、组织持久化
- 不同时维护两套同职能组件
- 不新增未经确认的 UI 框架或状态管理方案
- 不把大段业务流程持续堆进单个页面组件
- 不新增新的硬编码色值、断点、接口地址、状态码文案，除非已有规则要求并给出归档位置
